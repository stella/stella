# Plan: Internationalization (i18n)

Date: 2026-02-16

> **Note**: This plan was written for the initial i18next-based
> approach. The implementation pivoted to **use-intl** (ICU
> MessageFormat) for type-safe interpolation enforcement. See PR #276
> for the actual implementation.

## Goal

Add internationalization to Stella so the UI and API error messages
can be displayed in the user's preferred language. Initial target
locales: English (source), Czech, Polish, Hungarian, Slovak, German.

## OSS Research

Surveyed i18n implementations across several open-source projects to
inform our approach:

| Project          | Library                       | Namespaces            | Key style                  | Type safety                  | Languages | Translation platform |
| ---------------- | ----------------------------- | --------------------- | -------------------------- | ---------------------------- | --------- | -------------------- |
| **Cal.com**      | i18next + react-i18next       | Single (`common`)     | Flat `snake_case`          | Partial                      | 38        | Lingo.dev (AI)       |
| **Twenty** (CRM) | Lingui v5                     | Single per app        | Source strings as IDs (PO) | Macro-based                  | 31        | Crowdin              |
| **Plane**        | Custom (`intl-messageformat`) | Multi-file per domain | Nested `snake_case`        | Weak                         | 19        | None                 |
| **Rallly**       | i18next + react-i18next       | Multi-namespace       | Flat `camelCase`           | Strong (module augmentation) | 30        | Crowdin              |

**Key patterns we're adopting:**

- i18next + react-i18next (Cal.com, Rallly): battle-tested,
  excellent Slavic plural support, huge ecosystem.
- Feature-based namespaces (Plane, Rallly): scales better than
  cal.com's single 4,605-key file. Keeps translations colocated
  with features.
- `camelCase` keys with dot-path nesting (Rallly style): aligns
  with our TypeScript conventions.
- Full type generation via `i18next-resources-for-ts` + i18next
  module augmentation (Rallly pattern): compile-time safety
  that cal.com lacks.
- `sync-i18n` for multi-language file consistency: ensures all
  locales have the same key structure, flags gaps immediately.
- Lingo.dev CI for automated AI translation (Cal.com): fast,
  free for OSS; supplement with Crowdin later if human
  translators are needed (Twenty, Rallly both use Crowdin).
- `i18n-unused` for dead key detection (Cal.com).
- `_one`/`_few`/`_many`/`_other` plural suffixes: required for
  Czech, Polish, Slovak (3+ plural forms). Cal.com only uses
  `_one`/`_other` which is insufficient for Slavic languages.
- Terminology guide (`TERMINOLOGY.md`): strict rules for
  action verbs, legal terms, and consistency across locales.

## Design Decisions

- **`i18next` + `react-i18next` on the frontend.** Proven at
  scale (Cal.com: 38 locales, Rallly: 30 locales). Unlike Lingui
  (Twenty's choice), i18next uses JSON (not PO files), which is
  simpler for our workflow. Unlike Plane's custom solution, we
  get pluralization, interpolation, and tooling for free.

- **`i18next` standalone on the backend.** Same library, same
  translation files, consistent key names across the stack. API
  error messages that reach the user (toast messages, validation
  errors) will be translated server-side using the request locale.
  Cal.com uses the same pattern with `getTranslation(locale)`.

- **Feature-based namespaces, nested `camelCase` keys.** Unlike
  cal.com's single flat file (which becomes unwieldy at scale), we
  split by feature: `common`, `auth`, `workspaces`, `organization`,
  `errors`. Keys use nested `camelCase` matching our TS conventions:
  `auth.otp.sendFailed`, `workspaces.createFailed`. Rallly and
  Plane both use multi-namespace approaches successfully.

- **Locale stored on the user model, not in the URL.** Stella is a
  workspace app, not a content site; locale is a user preference.
  No `/en/workspaces` URL segments. Cal.com does the same: URLs
  are locale-agnostic, the language follows the user's setting.

- **Locale detection: user profile → `Accept-Language` → `"en"`.**
  Authenticated users get their stored preference. Anonymous users
  (login page, OTP page) fall back to browser language detection
  via `i18next-browser-languagedetector`. Same cascade as Cal.com.

- **Full type safety via generated types.** Use
  `i18next-resources-for-ts` to generate TypeScript interfaces
  from the English JSON, then augment i18next's `CustomTypeOptions`
  so `t("typo")` is a compile-time error. Rallly uses the same
  module augmentation pattern. Cal.com's partial typing is
  insufficient; Twenty's Lingui macros achieve similar safety but
  with a different DX.

- **Lingo.dev for automated translation CI, with Crowdin as a
  future option.** Cal.com uses Lingo.dev (free for OSS, AI-powered,
  fast). Twenty and Rallly use Crowdin (better for human translator
  workflows). Start with Lingo.dev; reassess if quality is
  insufficient for legal terminology.

- **Separate git worktree.** Work on a `feat/i18n` branch in a
  dedicated worktree to avoid conflicting with ongoing development.

## Scope

**In scope:**

- i18next setup for frontend (React provider, `useLocale` hook)
- i18next setup for backend (Elysia middleware, `getTranslation`)
- Translation file structure with feature-based namespaces
- English source string extraction from all existing components
- `locale` column on the `user` table (DB schema change)
- Locale detection middleware (browser language for anon users)
- Locale switcher in user settings (or sidebar footer)
- Type generation via `i18next-resources-for-ts`
- `sync-i18n` for multi-language file consistency
- Lingo.dev CI workflow for automated translations
- `i18n-unused` for dead key detection
- Pluralization with full Slavic plural rules
  (`_one`/`_few`/`_many`/`_other`)
- Date/number formatting with `Intl` APIs using the active locale
- `TERMINOLOGY.md` with translation guidelines per language
- RTL preparation (structural, not adding RTL locales yet)

**Out of scope:**

- URL-based locale routing (`/en/`, `/cs/` path prefixes)
- Legal terminology glossaries (future, per-jurisdiction)
- AI chat system prompt localization (separate effort)
- Translating AI-generated content (extraction results, etc.)
- Self-hosting locale override mechanisms
- Adding locales beyond the initial six
- Crowdin integration (future; start with Lingo.dev)

## Implementation

### 1. Git worktree setup

```bash
git worktree add ../stella-i18n feat/i18n
```

### 2. Dependencies

```bash
# apps/web
pnpm add i18next react-i18next i18next-browser-languagedetector
pnpm add -D i18next-resources-for-ts i18next-scanner

# apps/api
pnpm add i18next

# root (devDependencies)
pnpm add -D i18n-unused sync-i18n
```

### 3. Translation file structure

```
apps/web/src/translations/
├── locales/
│   ├── en.json          ← source of truth
│   ├── cs.json
│   ├── de.json
│   ├── hu.json
│   ├── pl.json
│   └── sk.json
├── i18n.ts              ← i18next init + config
├── use-locale.ts        ← custom useLocale() hook
├── i18n-provider.tsx    ← React context provider
└── TERMINOLOGY.md       ← translation guidelines
```

Each locale file uses nested namespaces:

```json
{
  "common": {
    "signOut": "Sign Out",
    "tryAgain": "Try again",
    "somethingWentWrong": "Something went wrong"
  },
  "auth": {
    "sendVerificationCode": "Send verification code",
    "sendFailed": "Failed to send verification code",
    "otp": {
      "enterCode": "Enter verification code"
    }
  },
  "workspaces": {
    "title": "Workspaces",
    "createNew": "Create new workspace",
    "createFailed": "Failed to create workspace",
    "createdAt": "Created at {{date}}",
    "limitReached": "Workspaces limit reached"
  },
  "organization": {
    "title": "Organization",
    "members": "Members",
    "invitations": "Invitations",
    "memberRemoved": "Member removed",
    "invitationCanceled": "Invitation canceled",
    "roles": {
      "owner": "Owner",
      "admin": "Admin",
      "member": "Member"
    }
  }
}
```

Pluralization example (Czech `cs.json`):

```json
{
  "common": {
    "day_one": "{{count}} den",
    "day_few": "{{count}} dny",
    "day_many": "{{count}} dní",
    "day_other": "{{count}} dní"
  }
}
```

### 4. Frontend i18n initialization

`apps/web/src/translations/i18n.ts`:

- Import all locale JSONs statically (with 6 locales, total
  bundle size is acceptable; lazy-load if we grow beyond ~10).
- Init i18next with `react-i18next` and
  `i18next-browser-languagedetector`.
- Configure `supportedLngs`, `fallbackLng: "en"`,
  `interpolation: { escapeValue: false }`.
- Export `supportedLanguages` array and `defaultLanguage` const.

### 5. React integration

- `apps/web/src/translations/i18n-provider.tsx` — wraps
  `I18nextProvider`, syncs locale from user session. Watches
  for `uiLanguage` changes and calls `i18n.changeLanguage()`.
- `apps/web/src/translations/use-locale.ts` — custom hook
  returning `{ t, i18n, locale }` with proper typing.
- Provider mounted in `apps/web/src/routes/__root.tsx`,
  wrapping `<Outlet />`.
- All components consume translations via `useLocale()`.

### 6. Backend i18n

- `apps/api/src/lib/i18n.ts` — server-side i18next instance.
  English bundled via `import`. Other locales loaded from a
  shared `translations/locales/` path (symlink or direct
  reference to the web app's locale files).
- `apps/api/src/middleware/locale.ts` — Elysia derive that
  reads locale from: (1) authenticated user's `locale` column,
  (2) `Accept-Language` header, (3) default `"en"`. Sets
  `context.locale` for downstream handlers.
- Handlers use `const t = getTranslation(context.locale)` for
  user-facing error messages.

### 7. DB schema change

`apps/api/src/db/auth-schema.ts` — add to `user` table:

```typescript
locale: text("locale").default("en").notNull(),
```

Migration via `bun run db:push`.

### 8. Locale switcher UI

- Add locale selector to sidebar footer (dropdown or popover).
- Uses `Intl.DisplayNames` to show language names in their
  native script: "Čeština", "Deutsch", "Magyar", "Polski",
  "Slovenčina".
- Mutation updates `user.locale` via API, then calls
  `i18n.changeLanguage()` client-side.

### 9. Date and number formatting

- Replace hardcoded `new Intl.Locale("en-GB")` in
  `apps/web/src/routes/_protected.workspaces/index.tsx` with the
  user's active locale.
- Create `useFormatDate()` and `useFormatNumber()` utilities
  that read locale from context and return formatted values via
  `Intl.DateTimeFormat` / `Intl.NumberFormat`.

### 10. Type safety

Two-layer approach:

1. `i18next-resources-for-ts` generates an interface from the
   English JSON:

   ```bash
   pnpm i18n:typegen
   ```

2. Module augmentation in `apps/web/src/types/i18n.d.ts`:

   ```typescript
   import type Resources from "./i18n-resources.gen";

   declare module "i18next" {
     interface CustomTypeOptions {
       resources: Resources["en"];
     }
   }
   ```

This gives autocomplete on `t()` calls and compile-time errors
for invalid keys.

### 11. CI tooling and scripts

`package.json` scripts:

```json
{
  "i18n:typegen": "i18next-resources-for-ts interface -i apps/web/src/translations/locales -o apps/web/src/types/i18n-resources.gen.ts",
  "i18n:sync": "sync-i18n --files apps/web/src/translations/locales/*.json --primary en --languages en cs sk pl hu de --space 2",
  "i18n:check": "pnpm i18n:sync -- --check",
  "i18n": "pnpm i18n:sync && pnpm i18n:typegen"
}
```

- `.github/workflows/i18n.yml` — Lingo.dev workflow (runs on
  push to main, opens PR with translations).
- `i18n.json` at repo root — Lingo.dev config (source locale,
  target locales, file paths).
- `i18n-unused` config to detect dead keys.

### 12. Terminology guide

`apps/web/src/translations/TERMINOLOGY.md`:

- Action verb conventions (create/delete/remove/update) per
  language, with examples.
- Legal terminology consistency rules.
- Pluralization rules summary for each target language.
- Date/number format expectations per locale.

### 13. String extraction

Systematic pass through all existing components and handlers to
replace hardcoded strings with `t("key")` calls. Priority order:

1. Navigation labels (sidebar, breadcrumbs)
2. Button text and form labels
3. Toast/error messages (frontend mutations)
4. API error messages (handlers)
5. Dialog titles and descriptions
6. Placeholder text and defaults
7. Validation messages
8. Role labels and other constants

## Test Cases

- English renders correctly (no regressions after extraction)
- Locale switching updates all visible text immediately
- Non-English locale loads correctly (e.g., Czech)
- Fallback to English for missing keys in a target locale
- Anonymous users see browser-detected language on auth pages
- `Accept-Language` detection works for unauthenticated API calls
- Plural forms work correctly for Czech and Polish (3+ forms)
- Date/number formatting respects active locale
- Type safety catches invalid translation keys at compile time
- `i18n:check` script detects missing keys across locales
- `i18n-unused` detects dead keys
- Locale preference persists across sessions (stored in DB)

## Open Questions

- **Where should the locale switcher live?** Sidebar footer
  dropdown vs. a dedicated user settings page. Leaning sidebar
  footer for discoverability.
- **Should API error messages be translated server-side or should
  the frontend map error codes to translated strings?** Server-side
  is simpler (Cal.com does this); client-side mapping is more
  flexible. Leaning server-side.
- **Should the chat system prompt be localized?** The AI might
  respond in the user's language regardless, but the system prompt
  sets expectations. Deferred for now.
- **When to move from Lingo.dev to Crowdin?** If we need
  professional human translators for legal terminology accuracy,
  Crowdin is the natural next step (Twenty and Rallly both use it).
